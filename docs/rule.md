# 🎮 少数派デスストーリー ルール仕様 v3（docs/rule.md）

---

## 📌 ゲーム概要

AIが生成する「身内ネタ入りストーリー」に対して、プレイヤーは毎ターン3択から1つを選ぶ。
選択肢ごとに「イベント効果（お金の増減）」が発生し、
**少数派（最も選ばれなかった選択肢）を選んだプレイヤーが追加ボーナス**を得る。

ゲーム終了時に**最も多くのお金を持っているプレイヤーが勝者**。
人生ゲームのように、運と読み合いで資産が大きく変わる。

---

## 👥 プレイ人数

- 3〜20人（推奨：8〜15人）

> 20人規模ではターン内の集計・演出に時間がかかるため、制限時間を短めに設定することを推奨。

---

## 💰 初期ステータス

- 所持金：¥1,000,000（100万円）
- 状態：全員「生存（現役）」

---

## 🎭 身内ネタストーリー生成

### コンセプト

ゲーム開始前に「身内ネタ情報」を登録しておき、AIがそれを自然にストーリーへ織り込む。
外部APIには**キーワードのみ**送信されるため、プライバシーリスクを最小化できる。

### 登録できる情報

```ts
type InsideJokeConfig = {
  playerNames: string[]; // 例: ["たぬき", "さくら", "ひろ"]
  keywords: string[]; // 例: ["深夜のラーメン", "例の失敗プレゼン", "謎の合言葉"]
  location: string; // 例: "大阪の居酒屋"
  theme: string; // 例: "会社の飲み会サバイバル"
  tone: "comedy" | "drama" | "chaos";
};
```

### AI生成の方式（2択）

| 方式                          | メリット                           | デメリット                                  |
| ----------------------------- | ---------------------------------- | ------------------------------------------- |
| **A：Claude API（推奨）**     | 高品質な日本語、自然なネタ組み込み | 外部API利用（キーワードのみ送信）           |
| **B：Ollama（ローカル完結）** | 完全オフライン、プライバシー最高   | 日本語品質やや低下、GPU推奨（llama3.1など） |

> Ollamaを使う場合は `ollama run llama3.1` 等を事前にセットアップ。
> 品質重視なら Claude API + キーワード限定送信が現実的。

## 🧠 コア仕様

### 選択肢構造

```ts
type Choice = {
  id: string;
  text: string; // ストーリーに沿った選択肢文
  riskLevel: "low" | "medium" | "high";
  mainEffect: MoneyEffect; // 全員（多数派）に適用
  minorityBonus: MoneyEffect; // 少数派のみ追加適用
};

type MoneyEffect = {
  type: "gain" | "lose" | "gamble" | "event";
  amount?: number; // 固定額
  rate?: number; // 変動率（例：0.2 = 現在の所持金の20%）
  description: string; // 演出テキスト
};
```

---

### 効果タイプ詳細

| タイプ   | 説明                                 | 例                                    |
| -------- | ------------------------------------ | ------------------------------------- |
| `gain`   | 固定額を獲得                         | +¥100,000                             |
| `lose`   | 固定額を失う                         | -¥150,000                             |
| `gamble` | 確率でプラス額が変動（マイナスなし） | 50%で+¥400,000 / 50%で+¥300,000       |
| `event`  | 所持金の割合で増減                   | 所持金の30%を失う（格差が広がる効果） |

> `event`タイプは「今持っているほど失う額が大きい」ため、1位のプレイヤーに重くなる自然な逆転要素になる。

---

### 設計ルール

- 3択は「低リスク / 中リスク / 高リスク」の構成
- 低リスクは安定小リターン、高リスクは大きく動く
- **明確な正解が存在しない**設計（状況・読み合いによって最適解が変わる）
- 少数ボーナスを含めても「圧倒的一択」にならないよう注意

## 🔄 ゲームの流れ

### 1. ゲーム開始

- GM（ゲームマスター）がルームを作成
- 身内ネタ情報を登録
- プレイヤーが参加コードでルームへ参加
- **GMはプレイヤーとして参加可能**（後述のGM制約あり）

---

### 2. ターン進行

#### ① ストーリー提示

- AIが身内ネタを織り込んだストーリーを生成
- 各選択肢に「リスクレベル」「メイン効果」「少数ボーナス」を表示

---

#### ② 選択フェーズ

- 各プレイヤーは1つ選択（制限時間：推奨15〜20秒）
- 未選択の場合：ランダム選択 or 未選択扱い（GM設定可）
- 選択中は他プレイヤーの選択は**非表示**

---

#### ③ 集計

- GMの見え方：**「各選択肢の合計人数のみ」**表示（誰が何を選んだかは不明）
- 一般プレイヤーの見え方：何も見えない（結果待ち）

---

#### ④ 判定

- 最も選択人数が少ない選択肢 → 「少数派」
- それ以外 → 「多数派」
- 複数の選択肢が同数最少 → **全て少数派として扱う**

---

#### ⑤ 効果適用

| 区分   | 適用効果                  |
| ------ | ------------------------- |
| 少数派 | メイン効果 + 少数ボーナス |
| 多数派 | メイン効果のみ            |

##### 逆転補正（ビリから立て直せる仕組み）

所持金が低いプレイヤーほど少数ボーナスが強化される：

| 所持金（現在）    | 少数ボーナス倍率 |
| ----------------- | ---------------- |
| ¥500,000以上      | ×1.0             |
| ¥200,000〜499,999 | ×1.3             |
| ¥199,999以下      | ×1.6             |

---

#### ⑥ 結果表示（全員・ログ画面に公開）

- ストーリー演出
- **誰が何を選んだか全員公開**（次ターンの読み合い材料）
- 所持金変動のアニメーション
- 脱落者表示（¥0以下）

---

## 📺 ログ画面（大画面映写用）

別URL（例：`/game/{roomId}/log`）で独立したログ画面を提供する。
**会場の大画面やプロジェクターに映しておくことを想定。**

### 表示内容

| 項目               | 表示形式                           |
| ------------------ | ---------------------------------- |
| 現在の順位         | リアルタイム資産ランキング         |
| 今ターンの選択結果 | 誰が何を選んだか（アイコン＋名前） |
| 各選択肢の選択人数 | 結果発表後に表示                   |
| 効果の演出テキスト | AIが生成したフレーバーテキスト     |
| ターン履歴         | 直近3〜5ターン分のサマリー         |

> このログ画面はGMが選択内容を見るためのものではなく、
> **全員が楽しむための実況演出画面**として設計する。
> GMは専用のGM画面（`/game/{roomId}/gm`）を使う。

---

## 👑 GM仕様

### GMプレイヤー参加時の制約

| 権限                       | GM（プレイヤー参加時）        |
| -------------------------- | ----------------------------- |
| 選択肢への投票             | ✅ 可能                       |
| 各選択肢の選択人数（集計） | ✅ 閲覧可                     |
| 誰が何を選んだか           | ❌ 不可（結果発表まで非表示） |
| 決着モードのON/OFF         | ✅ 可能                       |
| ターンスキップ             | ✅ 可能                       |

> GMが集計数を見られるのは「進行上の確認」のため（全員投票したか等）。
> 心理的有利にはならない設計とする。

### GM専用画面（`/game/{roomId}/gm`）

- 各選択肢の投票状況（人数のみ、リアルタイム）
- 未投票プレイヤーの確認
- ターン操作（次へ進む・延長・スキップ）
- 決着モードのON

---

## ☠️ 脱落条件

- 所持金が¥0以下で**破産**（脱落）
- 脱落後は観戦モード（ログ画面のみ参加可）

> ※ 破産者が出てもゲームは継続。最後の1人 or ゲーム終了条件まで進める。

---

## 🏁 ゲーム終了条件

### 設定ターン数で終了（推奨）

- GMが事前にターン数を設定（推奨：8〜15ターン）
- 終了時に**最も所持金が多いプレイヤーが勝者**

### 生存者1人で終了

- 全員が破産し1人だけ残った場合 → 即時勝者

### GM決着モード（時短）

- GMが任意タイミングでON
- 次ターン終了時に**最高所持金プレイヤーが勝者**

---

## ⚖️ 特殊ルール

### 全選択肢同数の場合

- 全員に「場のイベント」を適用（全員-¥50,000 などGM設定可能）

### 🟡 特殊イベント：「迷ったらYES」

通常の3択ターンとは別に、GMが任意のタイミングで発動できる特殊イベント。
**選択肢はYES / NOの2択のみ。**

#### 質問の性質

- AIが「迷う絶妙な質問」を生成する
- 例：「居酒屋で上司に『もう一軒行くぞ！』と言われたら？」
- 例：「謎のLINEグループに招待された。参加する？」
- 例：「全員参加の早朝マラソン企画が立ち上がった。名前を連絡先に書く？」
- ポイント：**常識的に断りたい / でもYESが正解かもしれない** という絶妙なラインの質問

---

#### 効果ルール（通常の少数派ロジックとは独立）

| 選択 | 区分   | 効果                                                    |
| ---- | ------ | ------------------------------------------------------- |
| YES  | 多数派 | +¥100,000（社訓を体現した全員へのご褒美）               |
| YES  | 少数派 | +¥250,000（少数でも信念を貫いた大ボーナス）             |
| NO   | 多数派 | -¥150,000（社訓を無視した代償）                         |
| NO   | 少数派 | -¥150,000（**少数派でもボーナスなし**、どう転んでも損） |

> **社訓「迷ったらYES」を知っているのにNOを選ぶ = 言い訳できない自業自得。**
> NOに少数ボーナスが存在しないのはこのゲーム唯一の例外ルール。

---

#### 発動タイミング

- GMが手動でトリガー（推奨：中盤〜終盤に1〜2回）
- 連続発動不可（2ターン以上間隔を空ける）
- 事前予告なし（突然始まる）

---

#### ストーリー演出

- 通常ターンとは異なる演出（例：画面が黄色くなる、BGM変化）
- 結果発表時に「社訓を守った人 / 裏切った人」を全員に公開
- ログ画面にも大きく表示（会場が盛り上がるポイント）

---

### 1人だけ選択した場合

- 完全少数派扱い → 少数ボーナス×2

### 20人規模でのターン運用

- 制限時間は短め（15秒）を推奨
- AI生成は投票フェーズ中に並行して次ターン分を先読み生成しておく

---

## 🎲 効果設計ガイドライン

### 基本数値（推奨・初期所持金¥1,000,000基準）

| リスクレベル | メイン効果           | 少数ボーナス                                          |
| ------------ | -------------------- | ----------------------------------------------------- |
| 低           | -¥50,000〜+¥50,000   | +¥100,000〜¥150,000                                   |
| 中           | -¥150,000〜+¥100,000 | +¥150,000〜¥250,000                                   |
| 高           | -¥300,000〜+¥200,000 | `gamble`系（+¥300,000〜400,000 確定、振れ幅のみ抽選） |

### 人生ゲームっぽいイベント例

- 「深夜のラーメンに誘われた → 行く（出費）/ 断る（睡眠OK）/ 奢る（大出費）」
- 「例の失敗プレゼン動画が発掘される → 笑う / 見ないふり / 本人に送る」
- 「飲み会の二次会 → 参加（出費）/ 先に帰る / 幹事に立候補（大出費 or 大評価）」

## 🎭 ストーリーモード仕様

> 通常モードとは独立した、GMが手書きシナリオを使うモード。  
> AIによる自動生成は行わない。シナリオはコードに静的定義。

---

### 概要

- ターン数：5〜7ターン（シナリオ定義による）
- 中心人物：**1位（最多所持金プレイヤー）** と **その他全員**
- エンディング：3種類（ハッピー / ノーマル / バッド）
- エンド決定方式：各ターンの選択に設定された**ストーリースコア**の累計で決定

---

### ストーリーターンの2フェーズ構造

```
Phase 1: first_selecting（1位専用）
  - 1位のみが専用の質問に回答
  - その他のプレイヤーは待機（見るだけ）

Phase 2: others_selecting（その他全員）
  - 1位の回答に応じて表示される選択肢が変わる
  - 1位以外の全員が投票
  - 1位は待機

→ Phase 1 + Phase 2 の組み合わせがストーリースコアに加算される
```

---

### データ構造

#### ストーリーターン定義

```ts
type StoryTurnDefinition = {
  turnIndex: number; // 0始まり
  background: string; // ターン開始時に全員に見えるナレーション

  // Phase 1: 1位用
  firstScene: {
    question: string;
    choices: {
      id: string;
      text: string;
      storyScore: StoryScore; // このターンの累計スコアへの寄与
    }[];
  };

  // Phase 2: その他用（1位の選択肢IDをキーに分岐）
  othersScenes: {
    [firstChoiceId: string]: {
      question: string;
      choices: {
        id: string;
        text: string;
        moneyEffect: MoneyEffect; // お金の増減
        storyScore: StoryScore;
      }[];
    };
  };
};

type StoryScore = {
  happy: number;
  normal: number;
  bad: number;
};
```

#### ルームへの追加フィールド

```ts
// Room に追加
storyProgress?: {
  currentTurnIndex: number;
  accumulatedScore: StoryScore;   // 全ターン合計
  ending?: 'happy' | 'normal' | 'bad';  // 最終ターン後に確定
};
```

#### ターンモード・フェーズの追加

```ts
// 既存: "normal" | "yesno"
// 追加:
mode: "story";

// 既存: "selecting" | "resolving" | "resolved"
// 追加:
phase: "first_selecting" | // 1位が回答中
  "others_selecting" | // その他が回答中
  "resolved"; // 両フェーズ完了
```

---

### ターン進行フロー

```
1. GMが「ストーリーターン開始」ボタンを押す
2. ナレーションが全員に表示
3. [first_selecting] 1位に専用の質問が表示 → 回答
4. GMが「次のフェーズへ」を押す
5. [others_selecting] 1位の回答に応じた選択肢が全員に表示 → 投票
6. GMが「結果確定/公開」を押す
7. お金が変動、ストーリースコアが加算、結果が表示
8. 最終ターンのみ：スコア集計 → エンディング確定・表示
```

---

### エンディング決定

全ターン終了後、`accumulatedScore` の最大値に対応するエンドに確定。

```
happy > normal && happy > bad  → ハッピーエンド
bad > normal && bad > happy    → バッドエンド
それ以外                        → ノーマルエンド
```

---

### シナリオ管理

- シナリオはコード内の静的配列として定義（`src/story/scenarios/`）
- 差し替えはコードを編集するだけ
- 将来的に複数シナリオ対応可（ルーム作成時に選択）

---

## 🚀 拡張案

- 状態異常（ブラックリスト入り：1ターン少数ボーナス無効 など）
- アイテム（1回だけ使えるジョーカー：選択を後出しできる等）
- 役職システム（インフォーマー：自分の選択を1人に教えられる）
- 観戦者がリアクションで演出を盛れる（絵文字爆発など）
- AI がプレイヤーの選択傾向を学習してストーリーをいじってくる

---

## 🧠 設計思想

- **心理戦**：履歴・所持金・キャラクターを読んで選択を予測する
- **リスク管理**：安全策 vs 一発逆転の永遠のジレンマ
- **逆転性**：ビリほど有利な補正＋eventタイプで1位がリスクを負う
- **盛り上がり**：ログ画面で全員の選択が公開された瞬間の「えーーー！」を作る
- **身内ネタ**：知っているエピソードが出てくるたびにグルーブが上がる
